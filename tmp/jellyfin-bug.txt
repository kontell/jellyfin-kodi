● There are two separate issues here, and the answer to your question is clear from line 3:                                                                                                                            
                                                            
  The ffmpeg command (constructed by Jellyfin's C# server code):                                                                                                                                                       
  ffmpeg ... -f hls -hls_time 3 -hls_segment_type fmp4 \                                                                                                                                                               
    -hls_fmp4_init_filename "...-1.mp4" \                                                                                                                                                                              
    -hls_playlist_type event -hls_list_size 0 \                                                                                                                                                                        
    -y "/var/lib/jellyfin/transcodes/....m3u8"                                                                                                                                                                         
                                                                                                                                                                                                                       
  This outputs a media playlist directly — not a master playlist. ffmpeg's HLS muxer only writes #EXT-X-STREAM-INF with RESOLUTION in a master playlist (-master_pl_name). Jellyfin doesn't pass -master_pl_name in the
   command, so no master playlist is generated, and no resolution metadata ends up in the m3u8.                                                                                                                        
                                                                                                                                                                                                                       
  But Jellyfin doesn't serve this file directly anyway — the /videos/{id}/live.m3u8 endpoint is a dynamic manifest generated by Jellyfin's C# code (DynamicHlsHelper / DynamicHlsController in the Jellyfin server     
  source). It constructs the m3u8 response on the fly, listing segments as they become available. That code doesn't include #EXT-X-STREAM-INF or resolution info either.
                                                                                                                                                                                                                       
  So the chain is: Jellyfin server C# code → generates dynamic HLS manifest → omits resolution metadata → inputstream.adaptive can't tell the hardware decoder the dimensions → falls back to software decode → crash  
  on 4K.
                                                                                                                                                                                                                       
  The timestamp errors (lines 45-50) are a separate issue — that's ffmpeg's HLS muxer struggling with out-of-order DTS from the source IPTV stream during the copy/remux.                                              
   
  Neither issue is in jellyfin-ffmpeg specifically (it's stock ffmpeg behavior) — both stem from how Jellyfin's server constructs the HLS pipeline.
